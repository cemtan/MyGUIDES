filesystem-forget-and-delete-nv-data

Remove filesystems from the registry; delete their data in NVRAM

Syntax

filesystem-forget-and-delete-nv-data <filesystems-see-'man sd-spec'>

Description

The server maintains a list of filesystems in its registry. It uses this list to ensure that each filesystem always has the same device ID and is always bound to the same EVS.

If you physically move storage between servers or you accidentally use an external configurator, such as Storage Navigator or Hitachi Command suite, to delete system drives or remove their Lun mappings so that the server cannot contact them, the old server will keep the filesystems in its registry. They will appear on SMU screens and in the output of commands such as 'trouble' and 'filesystem-list-stored'. If too many redundant filesystems appear in the registry, they can prevent you from creating new filesystems, because the number of available device IDs is finite.

'filesystem-forget-and-delete-nv-data' removes one or more filesystems from the registry. Specify the instance names or device IDs of the filesystems you wish to forget.

All the file systems' NVRAM data will be forgotten if you use this command. It will become impossible to replay NVRAM data for the file systems. The command has a deliberately long and explicit name to warn you that it is destructive.

You can't forget a filesystem if it's still loaded. Use 'span-deny-access' if you want to stop using a span and all its filesystems on this cluster, but first read the man page to find out that command's other side effects. If you use 'span-deny-access', there is no need to use 'filesystem-forget-and-delete-nv-data' as well.

Examples

filesystem-forget-and-delete-nv-data Accounts Marketing ProjectX

Applies To

Cluster wide

See Also
