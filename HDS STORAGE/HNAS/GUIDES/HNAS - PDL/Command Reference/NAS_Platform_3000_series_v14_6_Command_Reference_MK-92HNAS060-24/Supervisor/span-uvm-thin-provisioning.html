span-uvm-thin-provisioning

Make the server treat a span as being on UVM if local SDs aren't on a DP-Pool (regardless of whether remote SDs use HDP or not)

Syntax

span-uvm-thin-provisioning <span-instance-name> [no|yes]

Description

The server's support for Hitachi Universal Volume Manager is described in the 'uvm' man page.

'span-uvm-thinly-provisioned' sets a flag that informs the server that a UVM-based span (or a single tier of a tiered UVM-based span) resides on two or more external HDP pools, and that some of the span's stripesets reside on one external HDP pool while one or more other stripesets reside on one or more other external HDP pools. (Every new span contains one stripeset; a further stripeset is added every time the span is expanded by 'span-expand' or an equivalent GUI screen.)

If the flag is set then, during a filesystem-expansion where two or more stripesets are available, the administrator will be obliged to choose a stripeset, and thus a remote HDP pool, via 'filesystem-expand --on-stripeset'. If only one stripeset is available for a given expansion, or if the span does not reside on UVM internal LUs, the flag set by this command has no effect.

DOWNGRADES

The flag set by 'span-uvm-thinly-provisioned' is stored in span Cod. Setting this flag requires a new span Cod schema (format). If you downgrade to release 12.7, the server will successfully load the Cod but ignore the flag. If you downgrade to release 12.6 or earlier, the server will be unable to load the Cod, and you will be unable to mount file systems. Therefore, use this command only when necessary, and turn off the flag before attempting to downgrade the code on the server.

If you have to downgrade to release 12.6 without first disabling the flag, your service provider will be able to use 'sd-back-up-cod' to take a single-SD Cod backup, use the Cod Converter from release 12.7 to convert it to an integrated Cod backup (and discard the new flag in the process), use 'span-restore-cod' to restore the integrated Cod backup, and finally use 'span-rewrite-cod' to update the Cod so that it loads next time you reboot the cluster.

Examples

server:$ span-uvm-thin-provisioning Accounts yes
Success
server:$ span-uvm-thin-provisioning Accounts
Enabled
server:$

Applies To

Cluster wide
