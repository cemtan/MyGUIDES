sd-group-import

Import SD groups from a text file into the registry

Syntax

sd-group-import [--unknown-sd-action (abort|reject-group|trim-group)] [--conflict-resolution (abort|reject-new-group|delete-old-group)] [--tolerate-empty | -e] <text-file-name>
Alias:       impsdg

Description

Imports system drive (SD) groups into the server's registry from a text file produced by the 'sd-group-export' command. See the 'sd-group' man page for background.

If you are using pssc or ssc, the file is read from the client computer; otherwise, it is read from the server's ssfs file system.

This command is most useful when storage is moved between clusters or stand-alone servers. It enables administrators to enter group information on one server or cluster and then move it to other servers or clusters without needing to retype it. It is not necessary to use this command to ensure that all the nodes in a cluster know about SD groups; the cluster distributes this knowledge automatically.

When moving storage between stand-alone servers or clusters, be aware that SD device IDs will usually change. At first sight, this can make it look as if groups have been imported incorrectly.

Releases older than 12.6 supported parallel SD groups. If the input file contains any parallel SD groups, they will not be imported, and a warning will be displayed at the console. Any serial SD groups, however, will be imported as normal.

The command has three options:

--unknown-sd-action, -u

Controls what happens if the import file mentions SDs that are not known to this server. The options are:

abort

Aborts without importing anything from the file.

reject-group

Does not import any group that includes an unknown SD, but imports other groups if possible.

trim-group

Removes unknown SDs from groups; imports any groups that contain at least one SD after unknown SDs have been removed. This is the default option.

--conflict-resolution, -c

Controls what happens if groups being imported include SDs that are already grouped on the server. The options are:

abort

Aborts without importing anything from the file. This is the default option.

reject-new-group

Existing groups on the server are left intact; groups are imported only if they do not conflict with existing groups.

delete-old-group

As far as possible, all groups in the file are imported; where conflicts arise, groups already in the registry are deleted.

--tolerate-empty, -e

Without this switch, the import fails if the text file contains no groups. With this switch, a text file containing no groups is silently accepted, and no groups are affected.

These options can be abbreviated.

Importing SD groups never deletes existing groups unless they conflict with the groups being imported. 'sd-group-import' tries to merge the contents of the text file with any existing groups, rather than replacing existing groups.

'sd-group-import' is subject to the same restrictions as 'sd-group-create', 'sd-group-add', 'sd-group-remove' and 'sd-group-delete': for example, you cannot create a group containing SDs that are used in an open span.

Examples

sd-group-import groups.txt


sd-group-import --unknown-sd-action trim-group --conflict-resolution abort groups.txt

Applies To

Cluster wide
