sd-allow-access

License (allow access to) one or more system drives

Syntax

sd-allow-access [--ignore-foreign] <system-drives-see-'man sd-spec'>

Description

INTRODUCTION

The server will not let you create a new span on a system drive (SD) or load an existing span from an SD until you have allowed access to the SD. Allowing access to an SD is often called licensing the SD.

'sd-allow-access' allows access to the SD so that the server or cluster can use it.

Licensing information is stored in the server's or cluster's registry. The same storage can (and usually will) be licensed on one cluster and not on others, even if all the clusters are physically connected to the same storage. However, all servers in the same cluster always agree about whether an SD is licensed.

Never allow access to an SD that is being used by another server or cluster or a foreign server.

Normally, 'sd-allow-access' checks the SD for foreign operating systems' signatures and aborts if any are found. This check helps to prevent it from overwriting other operating systems' data. The --ignore-foreign switch skips this check.

You can allow access to several SDs with a single command. See the 'sd-spec' man page.

'sd-allow-access' sets off a number of asynchronous commands. Scripts may need to use 'span-wait-for-loading' (with the appropriate command-line arguments) to wait for these processes to complete before issuing further commands. If you wish to monitor the progress of these processes without waiting for them, run 'sd-list-queues' and 'filesystem-list-queues'. Typically, when you are working interactively, the background processes run fast enough for monitoring to be unnecessary.

LICENSING HDP DP-VOLS

The check for foreign servers' signatures involves performing disk I/O. If you are licensing HDP DP-Vols and the HDP pool on which they reside has not yet started formatting, this I/O will fail. We recommend waiting until a few percent of the pool has been formatted before licensing new DP-Vols.

STORAGE-BASED MIRRORING

Some high-availability installations use storage-based mirroring; see the 'storage-based-mirror' man page. If one of the SDs in a mirror relationship has been unlicensed in order that the server will ignore primary--primary relationships, licensing an SD might make the cluster notice a primary-primary relationship, and so fail the span. The command will not let you license an SD if a span would fail as a result.

Examples

sd-allow-access 0-7,16-23,Accounts

Allows access to SDs 0 to 7 and 16 to 23, as well as all SDs used in the span called 'Accounts'.

Applies To

Cluster wide
