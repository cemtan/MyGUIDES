trace-list

List traces

Syntax

trace-list [{group}]

Description

There are several trace commands available:

trace-list trace-none trace-off trace-on

The trace commands control a diagnostic facility for enabling or disabling various streams of debugging output.

Each of these streams of debugging output, or "traces", has two names, a group name and an individual name within that group. There are often several traces with the same group name. There may be traces with the same individual names which are within different groups.

Each trace has an enabled or disabled state. Some traces default to being enabled but most are off by default. There is also a node-wide enabled or disabled setting, referred to as "globally enabled" or "globally disabled". When disabled, the node-wide setting prevents any traces from generating output, regardless of whether they are enabled at the per-trace level.

When enabled, the output goes to the dblog and any local console. The main difference between "trace" and "logtrace" is that logtrace buffers a certain amount of output even when the stream output is disabled, so you can use it to look into the past. Although trace and logtrace are similar to use in the source code, they are not interchangeable at runtime - you can't use the logtrace command to inspect the previous output of a trace object.

trace-list

With no arguments, the trace-list command lists the enabled traces' names, then the disabled traces, each sorted alpha-numerically by group then individual name.

trace-list <group>...

When trace-list is followed by one or more group names, the traces in those groups are listed, categorized by enabled state.

trace-on

With no argument, "trace-on" alters the node-wide setting.

trace-on <trace-specifier>...

When "trace-on" is followed by one or more trace specifiers, output for those traces is enabled. The trace specifiers can be individual trace names or group names or names of the form group:name.

trace-off

With no argument, "trace-off" alters the node-wide setting.

trace-off <trace-specifier>...

When "trace-off" is followed by one or more trace specifiers, output for those traces is disabled. The trace specifiers can be individual trace names or group names or names of the form group:name.

trace-none

"trace-none" disables all traces at the per-trace level, but "trace-off" with no arguments is usually a better choice because there's no way to restore the set of previously enabled traces.

Persistent configuration

While the settings previously described take immediate effect, they are volatile and node-specific.

"telcset traceon" can be used with a comma-separated list of trace names to enable traces before the console is available during boot.

"telcset traceoff" works similarly.

Examples

There is no standard, de jure or de facto, for the output formatting used with traces.

This is a typical trace-using session.

$ trace-list nfs mountd
Locally Enabled nfs Traces:
<none>
Locally Disabled nfs Traces:
nfs: nfsniopcache
nfs: nfssecurity
nfs: nfsv2requests
nfs: nfsv3requests
nfs: nfsverbose
Locally Enabled mountd Traces:
<none>
Locally Disabled mountd Traces:
mountd: mountderrors
mountd: mountdfileops
mountd: mountdmsgbwriter
mountd: mountdrequests
mountd: mountdverbose


Trace is globally enabled on this node.
$ trace-on nfs mountd
Enabled output for Trace 'nfs'
Enabled output for Trace 'mountd'


Trace is globally enabled on this node.
$
$ # perform test, see trace output
$
$ trace-off


Trace is globally disabled on this node.


$ trace-off nfs mountd
Disabled output for Trace 'nfs'
Disabled output for Trace 'mountd'


Trace is globally disabled on this node.
$ trace-on


Trace is globally enabled on this node.
$

Applies To

Cluster node
