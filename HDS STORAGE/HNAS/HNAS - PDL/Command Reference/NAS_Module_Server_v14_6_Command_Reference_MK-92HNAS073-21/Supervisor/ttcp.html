ttcp

run ttcp client

Syntax

ttcp operation
where 'operation' is one of:
-t [--bufalign|-A ##] [--buflen|-l ##] [--port|-p ##] [--format|-f [K|G|M|k|g|m]] [--npackets|-n ##] host
-r [--port|-p ##] [--format|-f [K|G|M|k|g|m]] [--stop|-q]

Description

The ttcp program measures the transfer rate on a TCP or UDP connection. In order to use ttcp, you need to use a partner machine. One end needs to be listening and the other end needs to transmit to the listening end. This does NOT provide realistic performance figures for the server and should only be used as a diagnostic tool to check connectivity.

Operations:

--receiver, -r

Start up the listener

--transmitter, -t

Start transmitting to the specified host

--buflen, -l <length>

Use length as the buffer length (default 8192, minimum 128)

--port, -p <port>

Use port instead of the default of 5001

--bufalign, -A <alignment>

Use alignment instead of 0

--format, -f (K|G|M|k|g|m)

Print the speed in KiB/s, GiB/s, MiB/s, Kibit/s, Gibit/s or Mibit/s respectively

--npackets, -n <number>

Send number of packets (default 2048)

--stop, -q

Stop the listener

Unlike other ttcp programs, when started with the --receiver switch, the command starts up a listening socket in the background and then returns control to the console.

Examples

Start listener:

ttcp -r

Stop listener:

ttcp -r -q

If you want to try ttcp out with a version of ttcp that runs on linux, http://ftp.arl.mil/pub/ttcp/sgi-ttcp.c (as of 2006-12-08) has been tested. To use that to transmit to an existing listener on 172.18.0.100:

wget http://ftp.arl.mil/pub/ttcp/sgi-ttcp.c
make sgi-ttcp
./sgi-ttcp -t -s 172.18.0.100


server$ ttcp -r -f m
ttcp-r: started TCP listener on port 5001
ttcp-r: started UDP listener on port 5001
server$ ttcp-r: 16777216 bytes in 4.82 real seconds = 26.54 Mibit/sec +++ (27.8 Mbit/s (27833156 bit/s))

If the server is acting as a receiver, then dblog | grep ttcp-r is useful to observe the results.

Traditional ttcp endpoints display throughput in binary rather than decimal units. To avoid confusion, this implementation displays network throughput in both binary and decimal units. See http://physics.nist.gov/cuu/Units/binary.html for more details.

For your convenience:

100 Mbit/s is approximately 95.4 Mibit/s

1 Gbit/s is approximately 954 Mibit/s

You will never see these rates on a single line because of various overheads. A typical network might have 1460 bytes of application TCP data per Ethernet frame, with an inter-frame time equivalent to 1538 bytes, making the transport less than 95% efficient, before any account is taken of latency and TCP window size effects.
