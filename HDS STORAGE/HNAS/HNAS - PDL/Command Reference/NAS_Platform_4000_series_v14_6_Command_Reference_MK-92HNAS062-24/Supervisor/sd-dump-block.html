sd-dump-block

Read and hexdump one block from a system drive

Syntax

sd-dump-block [--denied] [--raw] <sd-device-sd> (byte|block) <number>

Description

Reads a block of data from a system drive (SD). Prints a hex dump of the data.

This command reads data directly from the storage, bypassing the fast path and all the caches in the server.

You can specify the offset either as a block number or as a byte offset; so, assuming 512-byte blocks, 'sd-dump-block 0 block 64' and 'sd-dump-block 0 byte 32768' are exactly equivalent. You can also (as with many commands) specify numbers in hex by using a '0x' prefix; so 'sd-dump-block 0 byte 0x8000' is a third way to write the same command.

The --denied or -d switch enables the command to read from unlicensed system drives (those to which you have not enabled access). Without this switch, the command will reject unlicensed SDs. Never use this switch to dump data from an SD that is being used by another cluster, because some storage does not support concurrent access from multiple clusters.

The --raw switch bypasses the Slie emulation layer, using the SD's real status to determine whether it is possible to read from disk. Since Slie is not normally used in production systems, this switch rarely has any effect.

Examples

cluster1:$ sd-dump-block 0 block 64
0x00000000 : 00 C0 0B E5 00 00 00 00 00 00 00 00 00 00 00 00 : ................
0x00000010 : 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 : ................
0x00000020 : 00 C0 0B E5 00 00 00 04 00 00 00 00 00 00 00 00 : ................
0x00000030 : 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 : ................
..... same (for 28 more lines, 448 more bytes) .....
0x00000200 :                                                 :
cluster1:$
