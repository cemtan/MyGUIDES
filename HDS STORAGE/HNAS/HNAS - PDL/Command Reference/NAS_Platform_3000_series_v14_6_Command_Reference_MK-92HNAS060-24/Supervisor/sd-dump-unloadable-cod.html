sd-dump-unloadable-cod

Show all unloadable span and filesystem Cod on all system drives

Syntax

sd-dump-unloadable-cod [-t | --terse]

Description

This is a troubleshooting command. It's intended for use by your support provider and inside diagnostic email messages.

When the server encounters system drives (SDs) at start of day, or when you run 'sd-rescan-cod' at any time, the server loads the three kinds of Cod, as described in the 'cod' man page.

The 'sd-dump-unloadable-cod' command dumps all unloadable Cod in the same format as 'sd-dump-cod'. It proceeds in two stages. First, it finds all SDs that have unloadable span Cod, and dumps that span Cod. For example, if span Foo uses SDs 1, 2 and 3, but SD 2's Cod has been damaged, the command will dump Cod from (at most) SDs 1 and 3. It's important to understand that if Cod has been destroyed on all visible SDs in a span, the server won't know that there's a damaged span, and 'dump-unloadable-cod' won't list any span Cod. Spans that contain just a single SD are unlikely ever to be listed.

In its second stage, 'sd-dump-unloadable-cod' finds each loaded span with an unloadable filesystem catalog or chunk table and, from each SD in the span, dumps whichever data structure won't load.

The --terse switch dumps the chunk table in a more concise format than normal. Other types of Cod are unaffected.

'sd-dump-unloadable-cod' makes no attempt to load Cod: it merely shows Cod that was unloadable last time the server tried to load it. For that, see 'sd-rescan-cod'.

It's not necessary to run 'sd-dump-unloadable-cod' and scour its output in order to find discrepancies between SDs. When the server finds a Cod discrepancy, it logs an event, specifying the SDs whose Cod disagrees. For filesystem Cod, you can also run 'span-list-filesystem-cod' for a concise table of Cod versions on all SDs in a loaded span.

Examples

sd-dump-unloadable-cod

Applies To

Cluster wide
