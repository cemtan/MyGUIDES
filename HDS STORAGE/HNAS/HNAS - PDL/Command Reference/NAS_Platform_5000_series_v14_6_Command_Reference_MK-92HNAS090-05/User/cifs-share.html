cifs-share

Manipulate CIFS shares on real file systems, cluster namespaces, or EVS namespaces

Syntax

cifs-share operation
where 'operation' is one of:
list [-v] [-n <num-entries>] [--target-label <namespace|file-system>] [<sharename>]
add [-e] [-S (disable|hidden|enable)] [-l] [-d] [-F] [-u] [--enable-abe] [-A] [-C (disable|enable)] [--encrypt-data] [-c <str>] [-o <cache-options>] [-t <access-configuration>] [--nodefaultsaa] [-h (off|ads|user|domainuser|unix) [-H <home-directory-root>]] [-R (do-not-transfer|transfer|use-fs-default)] <sharename> <namespace|file-system> <path>
del [--target-label <namespace|file-system>] <sharename>
delall [--delete-cns-shares] [--confirm]
mod [-e] [-s] [-S (disable|hidden|enable)] [-l | -m] [-d | -f] [-D | -F] [-u | -a] [--enable-abe | --disable-abe] [-A | -N] [-C (disable|enable)] [--encrypt-data | --no-encrypt-data] [-v <new-file-system>] [-p <path>] [-c <str>] [-o <cache-options>] [-t <access-configuration>] [--target-label <namespace|file-system>] [-h (off|ads|user|domainuser|unix) [-H <home-directory-root>]] [-R (do-not-transfer|transfer|use-fs-default)] <sharename>
check-access [--target-label <namespace|file-system>] <sharename> <client-ip-addr>
Alias:       cns-cifs-share

Description

The cifs-share command is used to manage CIFS shares on normal file systems, or cluster/EVS namespaces. A CIFS share makes a directory and everything below it available over CIFS.

In cluster/EVS namespaces, any path that does not traverse a link to a normal file system can be shared.

Usage:

cifs-share list [OPTION]... [<sharename>]

List the details of the share(s). If a sharename is specified, only details for the given share will be displayed. If neither a target-label nor a sharename is specified, all the namespace and filesystem shares in the context of the current EVS will be listed. In the context of the Admin Service, only cluster namespace shares will be displayed. Options are:

-v, --verbose

Display verbose information.

-n, --max <num-entries>

Lists at most num-entries shares.

--target-label <namespace|file-system>

Name of the file system or the cluster/EVS namespace in which to look for the share name. This option is valid only if a share name is provided. If this option is not provided we try to locate the share on both the namespace and the file systems that are visible in the context of the current EVS.

cifs-share add [OPTION]... <sharename> <namespace|file-system> <path>

Add a new CIFS share. Options are:

-e, --ensure

Ensure that the path exists. Create the path to the share if it doesn't already exist. Only applicable for CIFS shares for file systems.

-S, --snapshot-dirs

Sets the accessibility and visibility of the ~snapshot directory:

disable

Disable ~snapshot directory. This option should be selected for a share that includes a virtualization path. The virtualization-path-list command can be used to list virtualization paths.

hidden

Enable access to but hide ~snapshot directory.

enable

Enable access to and show ~snapshot directory. This is the default.

-l, --lowercaseoncreate

Convert the names of all files and directories created via this share to lower case.

-d, --dontfollowsymlinks

If symlinks are encountered when browsing this share, don't follow them automatically on the server. If client side symlink handling is enabled, this setting does not affect clients using the SMB2 or SMB3 protocols.

-F, --followglobalsymlinks

Allow clients that are connected to this share to follow global symlinks.

-u, --noscanforviruses

If virus scanning is enabled, don't scan files accessed via this share for viruses.

-c, --comment <comment>

Set the comment for this share. The comment is a free-form text field that can be used to describe the share.

-o, --cache-options <options>

Set the share's cache options to one of the following:

0

Manual local caching for documents (default)

4

Automatic local caching for documents

8

Automatic local caching for programs

12

Local caching disabled

--enable-abe

Enable Access-based Enumeration, which makes visible only those files or folders that the user has rights to access.

-A, --continuous-availability

Enable continuous availability for this share. Files that are opened using SMB3 on a continuously available share can survive network failures as well as cluster node failures. Continuous availability requires client assistance.

Continuous availability is not available on replication target file systems. It may be selected, but will be disabled until the file system is remounted as a read-write file system.

Note: continuous availability may impact performance and should only be enabled where it is required (e.g. Microsoft Hyper-v, Microsoft SQL Server).

-C, --shadow-copy-use

Controls the participation of this share in a ShadowCopy point-in-time set.

enable

Allow this share to be part of a ShadowCopy set. This is the default.

disable

Refuse any requests to make this share a part of a ShadowCopy set.

--encrypt-data

Turn on encrypted access. Encrypted access makes data transfers secure by encrypting data in-flight, to protect against tampering and eavesdropping attacks. Clients using the SMB2 protocol that do not support encrypted access are unable to connect to this share (but see smb3-reject-unencrypted-access-disable). Clients using the SMB1 protocol ignore this setting allowing unencrypted access. See smb-min-supported-version to disable SMB1 clients. Only clients using dialects >= SMB3.0 support encryption. See smb-max-supported-version and smb3-encryption-enable for more details. Note that this feature does not encrypt the data on the disk.

-t, --access-configuration <options>

Set the client access restrictions for this share. The default is '*(rw)', meaning that all clients have read and write access to this share.

See mount-point-access-configuration and umask- for more details.

--nodefaultsaa

By default, a new share gets a Share Access Authentication (SAA) entry that grants Full Control to Everyone. This switch prevents that entry from being created, which means it will be inaccessible until an SAA entry is added.

-h, --home-directory <option>

Set the share's home directory behaviour to one of the following:

off

Disable the home directory feature for this share. No home directories will be created for any users. This is the default if the option is not supplied.

user

A per-user home directory should be created, with the user's directory name being the same as their Windows username, converted to lower case.

domainuser

A per-user home directory should be created, with the user's directory name being the same as their Windows username, converted to lower case. The home directory will be created in a sub-directory named for the user's Windows domain. For example, a user "DOMAIN\John Smith" would result in a home directory of "domain\john_smith".

unix

A per-user home directory should be created, with the user's directory name being the same as their UNIX username, converted to lower case.

ads

Use the home directory information supplied by the Active Directory server for each user. If this option is selected, then the --home-directory-root option must not be specified.

-H, --home-directory-root <home-directory-path>

Per-user home directories will be created in home-directory-path, relative to the share root, which is specified with no leading '\'. If not specified, or if set to the empty string, user home directories will be created directly in the share root. By default, only one share per file system can be configured with home directories. The cifs-home-directory command can be used to relax this restriction, in which case great care must be taken not to configure conflicting home directories. For example, a share with a path of \home1 and a share with a path of \home2 would not cause a conflict, whatever home directory paths were configured. However, a share with a path of \ and a default home directory path would conflict with a share with a path of \dir and a default home directory path.

-R, --disaster-recovery-setting <option>

This setting only applies to shares that target WFS-2 file systems.

Sets whether the share should be brought online when the replication target of this share's file system is converted to read-write.

Defaults to 'use-fs-default' if this setting is not specified explicitly.

Set this option to one of the following:

transfer

Mark the share for transfer to the replication target of this share's file system, indicating that it should be brought online when the replication target is mounted read-write.

do-not-transfer

Do not transfer this share to the replication target.

use-fs-default

Use the file system default value to decide whether this share should be transferred to the replication target or not.

cifs-share del [OPTION]... <sharename>

Delete the given share. Options are:

--target-label <namespace|file-system>

Name of the file system or the cluster/EVS namespace in which to look for the share name. If this option is not provided we try to locate the share on both the namespace and the file systems that are visible in the context of the current EVS.

cifs-share delall [OPTION]

Delete all configured CIFS shares in both the file systems and the EVS namespaces (applies to EVSs configured with individual security setting only) visible in the context of the current EVS.

--confirm

Delete all the shares without prompting for further confirmation from the user.

--delete-cns-shares

Delete all CIFS shares in the cluster namespaces visible in the context of the current EVS as well. This only applies to EVSs configured with global security settings.

An EVS can be configured either with individual or global security setting.

WARNING: There exists only a single instance of a cluster namespace CIFS share, which is visible in the context of all EVSs configured with global security. Thus, deleting such a share using this option will delete the single instance and make it unavailable to *all* of the EVSs configured with global security settings (see 'evs-security list').

cifs-share mod [OPTION]... <sharename>

Modify an existing CIFS share. Options are:

--target-label <namespace|file-system>

Name of the file system or the cluster/EVS namespace in which to look for the share name. If this option is not provided we try to locate the share on both the namespace and the file systems that are visible in the context of the current EVS.

-e, --ensure

Ensure that the path exists. Create the path to the share if it doesn't already exist. Only applicable for CIFS shares for file systems.

-s, --saadefault

Add an 'Everyone, Full Control' SAA entry to this share.

-S, --snapshot-dirs

Sets the accessibility and visibility of the ~snapshot directory:

disable

Disable ~snapshot directory. This option should be selected for a share that includes a virtualization path. The virtualization-path-list command can be used to list virtualization paths.

hidden

Enable access to but hide ~snapshot directory.

enable

Enable access to and show ~snapshot directory.

-l, --lowercaseoncreate

Convert the names of all files and directories created via this share to lower case.

-m, --maintaincaseoncreate

Don't convert the names of files and directories created via this share to lower case, but maintain the case as provided by the client.

-d, --dontfollowsymlinks

If symlinks are encountered when browsing this share, don't follow them automatically on the server. If client side symlink handling is enabled, this setting does not affect clients using the SMB2 or SMB3 protocols.

-D, --dontfollowglobalsymlinks

Don't allow clients that are connected to this share to follow global symlinks.

-f, --followsymlinks

If symlinks are encountered when browsing this share, follow them automatically on the server. If client side symlink handling is enabled, this setting does not affect clients using the SMB2 or SMB3 protocols.

-F, --followglobalsymlinks

Allow clients that are connected to this share to follow global symlinks.

-u, --noscanforviruses

If virus scanning is enabled, don't scan files accessed via this share for viruses.

-a, --scanforviruses

If virus scanning is enabled, scan files accessed via this share for viruses.

-v, --new-file-system <file-system>

Change the file system to which the share points.

-p, --path <path>

Change the path to which the share points.

-c, --comment <comment>

Change the share's comment.

-o, --cache-options <options>

Change the share's cache options. See above for a list of values.

--enable-abe

Enable Access-based Enumeration, which makes visible only those files or folders that the user has rights to access.

--disable-abe

Disable Access-based Enumeration.

-A, --continuous-availability

Enable continuous availability for this share. Files that are opened using SMB3 on a continuously available share can survive network failures as well as cluster node failures. Continuous availability requires client assistance.

Continuous availability is not available on replication target file systems. It may be selected, but will be disabled until the file system is remounted as a read-write file system.

Note: continuous availability may impact performance and should only be enabled where it is required (e.g. Microsoft Hyper-v, Microsoft SQL Server).

-N, --no-continuous-availability

Disable continuous availability for this share.

-C, --shadow-copy-use

Controls the participation of this share in a ShadowCopy point-in-time set.

enable

Allow this share to be part of a ShadowCopy set.

disable

Refuse any requests to make this share a part of a ShadowCopy set.

--encrypt-data

Turn on encrypted access. Encrypted access makes data transfers secure by encrypting data in-flight, to protect against tampering and eavesdropping attacks. Clients using the SMB2 protocol that do not support encrypted access are unable to connect to this share (but see smb3-reject-unencrypted-access-disable). Clients using the SMB1 protocol ignore this setting allowing unencrypted access. See smb-min-supported-version to disable SMB1 clients. Only clients using dialects >= SMB3.0 support encryption. See smb-max-supported-version and smb3-encryption-enable for more details. Note that this feature does not encrypt the data on the disk.

--no-encrypt-data

Turn off encrypted access.

-t, --access-configuration <options>

Change the client access restrictions for this share.

See mount-point-access-configuration and umask- for more details.

-h, --home-directory <option>

Set the share's home directory behaviour. See above for a list of the values.

-H, --home-directory-root <home-directory-path>

Per-user home directories will be created in home-directory-path, relative to the share root, which is specified with no leading '\'. See above for details.

-R, --disaster-recovery-setting <option>

Sets the share's disaster-recovery behaviour. See above for a list of the values.

check-access [OPTIONS] <sharename> <client-ip-addr>

Reports the access configuration that applies to the specified client. IPv4 or IPv6 addresses can be used. Options are:

--target-label <namespace|file-system>

Name of the file system or the cluster/EVS namespace in which to look for the share name. If this option is not provided we try to locate the share on both the namespace and the file systems that are visible in the context of the current EVS.

'thin provisioned' refers to file system-level thin provisioning, as described in the 'filesystem-thin' man page. It is unrelated to the configuration of the block storage.

To create the directory \home\myuser on the file system fs0 and to create a CIFS share named myshare that makes it available over CIFS:

cifs-share add -e myshare fs0 \home\myuser

To modify the client access restrictions to share 'myshare' on 'fs0' so that only clients in the 10.1.1.* subnet have read-write access, whereas all other clients have read-only access:

cifs-share mod -t "10.1.1.*(rw), *(ro)" --target-label fs0 myshare

To check which access restriction matches a given client:

cifs-share check-access --target-label fs0 myshare 10.1.1.1
10.1.1.*        rw, can browse

To modify share 'myshare' to point at the directory \home\mynewuser on the file system fs3:

cifs-share mod -v fs3 -p \home\mynewuser --target-label fs0 myshare

To create a CIFS share named 'mycnsshare' that makes the root of the cluster namespace 'cns0' available over CIFS:

cifs-share add mycnsshare cns0 \

To modify the client access restrictions to share 'mycnsshare' so that only clients in the 10.1.1.* subnet have read-write access, whereas all other clients have read-only access:

cifs-share mod -t "10.1.1.*(rw), *(ro)" --target-label cns0 mycnsshare

To check which access restriction matches a given client:

cifs-share check-access --target-label cns0 mycnsshare 10.1.1.1
10.1.1.*        rw, can browse

To create a CIFS share named 'myevsnsshare' that makes the root of the EVS namespace 'evs1' available over CIFS:

cifs-share add myevsnsshare evs1 \

To modify the client access restrictions to share 'myevsnsshare' so that only clients in the 10.1.1.* subnet have read-write access, whereas all other clients have read-only access:

cifs-share mod -t "10.1.1.*(rw), *(ro)" --target-label evs1 myevsnsshare

To check which access restriction matches a given client:

cifs-share check-access --target-label evs1 myevsnsshare 10.1.1.1
10.1.1.*        rw, can browse

To modify the client access restrictions to share 'myevsnsshare' so that only clients in the fdca:f995:220a:a00::/64 IPv6 subnet have read-write access, whereas all other clients have read-only access:

cifs-share mod -t "fdca:f995:220a:a00::/64(rw), *(ro)" --target-label evs1 myevsnsshare

To check which access restriction matches a given IPv6 client:

cifs-share check-access --target-label evs1 myevsnsshare fdca:f995:220a:a00:1a03:73ff:fe30:3531
fdca:f995:220a:a00::/64	ro

To create a CIFS share and specify that home directories should be created in directory \dir\home\users in file system fs0 with the same name as a user's Windows username:

cifs-share add -h user -H home\users myshare fs0 \dir

To have the Active Directory server specify the user's home directory in the file system fs0:

cifs-share add -h ads myshare fs0 \dir

Applies To

Cluster wide or EVS
